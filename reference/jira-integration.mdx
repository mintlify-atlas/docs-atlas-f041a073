## Overview

Open Orchestra supports Jira as an alternative issue tracker for managing tickets, subtasks, and workflow execution. While Linear is the default, you can configure Open Orchestra to use Jira by editing the runtime configuration file.

## Prerequisites

<Info>
  Ensure you have admin access to your Jira project to create custom fields and manage issue types if needed.
</Info>

- Jira project access
- Jira MCP server installed and configured
- Valid Jira API credentials (API token or OAuth)
- Jira project key

## Setup

<Steps>
  <Step title="Install the Jira MCP server">
    Follow the Jira MCP server installation instructions for your agent runtime environment. The MCP server provides the communication layer between Open Orchestra agents and your Jira instance.
  </Step>

  <Step title="Configure orchestra-config.json">
    Edit the `orchestra-config.json` file in your repository root to specify Jira as your issue tracker:

    ```json orchestra-config.json
    {
      "issue_tracker": "jira"
    }
    ```

    If the file doesn't exist, it will be created automatically with Linear as the default when you run your first skill. Simply edit it afterward to switch to Jira.
  </Step>

  <Step title="Verify MCP availability">
    Before running any skill, ensure the Jira MCP server is available and can communicate with your Jira instance. If the configured issue tracker MCP is unavailable, skills will stop immediately and not proceed with tracker operations.
  </Step>
</Steps>

## Usage

### Creating tickets

All tracker-writing skills create issues in Jira using the configured MCP:

- **requirements-ticket-agent** creates parent issues with structured requirements
- **planning-agent** creates Technical Plan subtasks and implementation subtasks
- **qa-agent** creates QA plan subtasks

<Note>
  Subtasks in Jira must be linked to their parent issue. The MCP handles this relationship automatically.
</Note>

### Workflow labels

Open Orchestra uses Jira labels to track workflow stage completion:

**Completion labels:**
- `requirements-done`
- `planning-done`
- `qa-done`
- `implementation-done`
- `pr-published`
- `pr-reviewed`

**Open-question labels:**
- `open-requirements-questions`
- `open-planning-questions`
- `open-qa-questions`
- `open-implementation-questions`
- `open-pr-review-questions`

<Tip>
  These labels are automatically added by agents as they progress through workflow stages.
</Tip>

### Issue status transitions

Agents update Jira issue status as work progresses:

- **In Progress** — Set by planning-agent after planning is complete
- **In Review** — Set by implementation-agent after PR is published
- **Done** — Set by pr-review-agent when review is clean

<Warning>
  Ensure your Jira workflow includes these status values, or configure your Jira workflow to accept the status transitions used by Open Orchestra.
</Warning>

### Audit trail

Every tracker operation includes a skill version stamp for traceability:

```
Skill-Version: implementation-agent@0.0.1
```

All tracker-writing skills leave an explicit audit trail with:

- Rich artifact content in issue/subtask descriptions
- `Reasoning` section explaining agent decisions
- `References` section listing exact inputs used
- `Assumptions` section with unresolved unknowns
- Parent issue comment named `Execution-Trace` with actions, decisions, and blockers

## Handoff comments

Agents write handoff comments to Jira issues in a structured JSON format wrapped in `<!-- OPEN-ORCHESTRA-HANDOFF -->` markers. These comments enable efficient context passing between agents without reading full issue history.

<Note>
  Downstream agents read only the most recent handoff comment for stage-specific context, avoiding expensive full history reads.
</Note>

## Story points

Open Orchestra applies story point estimates to parent issues only:

- `story-point-2` — Very small scope, low risk
- `story-point-3` — Small scope, moderate complexity
- `story-point-5` — Medium scope or cross-module work
- `story-point-8` — Large scope with dependencies
- `story-point-13` — Very large/high uncertainty

<Info>
  Issues scored 8 or 13 automatically receive a `human-review-required` label.
</Info>

## Troubleshooting

<Warning>
  If the Jira MCP is unavailable, all tracker operations will be blocked and skills will stop immediately.
</Warning>

### Common issues

**MCP server not available**
- Verify the Jira MCP server is running
- Check your MCP server configuration and Jira instance URL
- Ensure network connectivity to Jira (especially for cloud instances)
- Verify firewall rules allow outbound connections

**Authentication errors**
- Verify your Jira API token or OAuth credentials are valid
- Check that credentials haven't expired
- Ensure the configured user has project access

**Permission errors**
- Verify your Jira user has permissions to create issues and subtasks
- Ensure you can add comments and update issue status
- Check project permissions for creating custom fields and labels

**Status transition failures**
- Verify your Jira workflow includes the required status values
- Check that your user has permission to perform the status transition
- Review workflow rules that may block automated transitions

**Missing labels**
- Open Orchestra creates workflow labels automatically
- If labels are missing, check that the agent completed its stage successfully
- Review the `Execution-Trace` comment for any reported errors

## Switching from Jira

To switch from Jira back to Linear, edit the `issue_tracker` value in `orchestra-config.json`:

```json orchestra-config.json
{
  "issue_tracker": "linear"
}
```

<Tip>
  No other configuration changes are required. Skills automatically use the configured tracker MCP.
</Tip>
