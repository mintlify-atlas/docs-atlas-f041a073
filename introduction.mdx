## What is Open Orchestra?

Open Orchestra is a skill pack for AI agents that orchestrates complete software delivery workflows. It takes informal user requests and transforms them into shipped code through a structured, auditable pipeline that mirrors agile development practices.

Instead of requiring pre-built architecture documentation that drifts out of sync, Open Orchestra agents read your live codebase directly. The planning agent navigates source files using search and file-reading tools to produce precise technical plans grounded in what the code actually looks like right now.

## How it works

The workflow runs as a pipeline of specialized agent skills:

<Steps>
  <Step title="Requirements gathering">
    `requirements-ticket-agent` converts informal requests into structured tickets with Context, Goal, Scope, Constraints, and Acceptance Criteria.
  </Step>
  
  <Step title="Triage and routing">
    `triage-agent` classifies work as trivial, standard, or complex and sets `skip_steps` guidance for downstream efficiency.
  </Step>
  
  <Step title="Technical planning">
    `planning-agent` reads the ticket, inspects the live codebase, and creates a Technical Plan subtask plus up to 8 implementation subtasks.
  </Step>
  
  <Step title="QA planning (parallel)">
    `qa-agent` creates ticket-native test cases from requirements and implementation subtasks. Runs in parallel without blocking.
  </Step>
  
  <Step title="Implementation and PR publish">
    `implementation-agent` executes subtasks in sequence, records build/lint outcomes, and publishes the PR in the same run.
  </Step>
  
  <Step title="PR review">
    `pr-review-agent` performs diff-first review with severity-ranked findings. Routes back to implementation when fixes needed.
  </Step>
</Steps>

## Core design principles

### The code is the architecture document

Rather than requiring pre-built architecture snapshots that drift out of date, `planning-agent` reads the ticket requirements, navigates the live codebase using search and file-reading tools, and produces a short, precise technical plan grounded in what the code actually looks like right now.

This means:
- No architecture initialization ritual before every ticket
- The coding agent always sees the real, current state of the code
- Less maintenance overhead; fewer stale context files confusing the agent
- Implementation subtasks reference actual file paths and existing patterns — not guesses from an outdated doc

<Tip>
  You can still create or maintain architecture docs. Pass them as optional context when invoking `planning-agent`, and the agent will read them alongside the live code.
</Tip>

### Tracker audit trail contract

All tracker-writing skills must leave an explicit audit trail. A run is incomplete unless all items below are present:

- Rich artifact content (issue/subtask body), not title-only updates
- `Reasoning` section that explains why the agent chose the proposed scope/plan
- `References` section listing exact inputs used (ticket fields, child tasks, files, PR links, commands when relevant)
- `Assumptions` section with unresolved unknowns and inferred decisions
- Parent issue comment named `Execution-Trace` with:
  - actions performed (in order)
  - key decisions
  - blockers/open questions
  - `Skill-Version` stamp

### JSON contract for deterministic chaining

In addition to readable markdown sections, each agent must emit machine-parseable JSON blocks for deterministic chaining and validation.

**Execution-Trace JSON contract:**

```json
{
  "agent": "planning-agent",
  "skill_version": "planning-agent@0.0.1",
  "timestamp_utc": "2026-02-22T10:30:00Z",
  "actions": ["read_parent_issue", "inspected_codebase", "created_technical_plan"],
  "decisions": ["split_api_and_ui_work", "deferred_non_blocking_cleanup"],
  "blockers": [],
  "open_questions": [],
  "references": ["LIN-123", "src/routes/v1/router.ts", "src/models/order.ts"]
}
```

**Workflow-Handoff JSON contract:**

```json
{
  "from_agent": "planning-agent",
  "to_agent": "implementation-agent",
  "status": "ready|blocked",
  "required_tags": ["planning-done"],
  "open_question_tags": [],
  "summary": "Technical plan posted. Implementation subtasks created and prioritized.",
  "artifacts": ["LIN-456", "LIN-457", "LIN-458"],
  "next_actions": ["implement_subtasks_in_order", "publish_pr_when_complete"]
}
```
